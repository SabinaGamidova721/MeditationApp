# ZenFlow — Local Storage, API Contract, Sync Strategy

## Обрана стратегія: Mix

У застосунку використовується змішана стратегія.

Медитації працюють як cache + refresh:

- спочатку дані читаються з локального сховища;
- після цього застосунок пробує отримати свіжі дані з API;
- якщо API відповідає успішно, локальне сховище оновлюється.

Сесії користувача працюють як offline-first:

- завершена медитація одразу зберігається локально;
- якщо інтернету немає, сесія залишається зі статусом `pending`;
- коли інтернет зʼявляється, застосунок пробує синхронізувати pending-сесії;
- після успішної синхронізації статус змінюється на `synced`.

## API-контракт

| Метод  | Шлях               | Що повертає               | Параметри / тіло                                      |
| ------ | ------------------ | ------------------------- | ----------------------------------------------------- |
| GET    | `/meditations`     | Масив медитацій           | Нічого                                                |
| GET    | `/meditations/:id` | Одну медитацію або `null` | `id` у path                                           |
| POST   | `/sessions`        | `{ success: true }`       | `id`, `meditationId`, `completed`, `duration`, `date` |
| DELETE | `/sessions/:id`    | `{ success: true }`       | `id` у path                                           |
| GET    | `/user`            | Дані користувача          | Нічого                                                |

## Локальне сховище

Для локального сховища використано AsyncStorage.

Реалізовано збереження для ключових сутностей:

- `Meditation`
- `Session`
- `User`

Для `Meditation` та `Session` реалізовано:

- зберегти;
- прочитати список;
- прочитати один обʼєкт;
- видалити.

Для `User` реалізовано:

- зберегти;
- прочитати;
- видалити.

Екрани списку та деталей читають дані зі сховища через Repository, а не напряму з hardcoded масиву.

## WebSocket / Real-time

У застосунку реалізовано real-time канал через `MockSocketManager`.

Оскільки реального WebSocket-сервера немає, використано заглушку, яка імітує сервер:

- при відкритті екрана `Live` встановлюється зʼєднання;
- стан змінюється: `Disconnected → Connecting → Connected`;
- кожні 4 секунди приходить тестова подія;
- події парсяться у модель `SocketEvent`;
- UI оновлюється через `useState` без перезавантаження екрана;
- при симуляції розриву зʼєднання стан переходить у `Reconnecting`;
- через 1 секунду відбувається автоматичне перепідключення.

Підтримані події:

| Подія            | Що робить                               |
| ---------------- | --------------------------------------- |
| `motivation`     | Показує мотиваційне повідомлення        |
| `new_meditation` | Додає нову медитацію в локальне сховище |
| `server_status`  | Показує статус сервера                  |
| `session_bonus`  | Додає бонусні хвилини користувачу       |
